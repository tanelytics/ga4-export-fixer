# Query Builder SQL Formatting

**Status**: Planned
**Priority**: P2 (Low)
**Estimated**: 0.5 days
**Dependencies**: None

## Problem Statement

The SQL generated by `queryBuilder()` in `utils.js` has inconsistent indentation that makes the output difficult to read. Three specific issues:

### 1. CTE bodies have no indentation

The `select`, `from`, `where`, and `group by` keywords inside a CTE sit at column 0, making it hard to see where one CTE ends and the next begins:

```sql
with event_data as (select
    concat(...) as session_id,
    ...
from
    `project.dataset.events_*`
where
    ...
),
    session_data as (select
    session_id,
    ...
```

### 2. Multi-line column values lose alignment

Helper functions (e.g., `fixEcommerceStruct`, `extractPageDetails`, `aggregateValue`) return multi-line SQL fragments with their own internal indentation. When `queryBuilder` inserts these at a column position, only the first line aligns — continuation lines stay at the helper's absolute indentation:

```sql
    (select as struct ecommerce.* replace(
    if(ecommerce.transaction_id <> '(not set)', ...) as transaction_id,
    if(
      event_name = 'purchase',
      coalesce(
        ...
```

The `if(` lines should be indented relative to the column position, not at column 0+helper offset.

### 3. Spurious blank lines between clauses

When a step has no `leftJoin`, `where`, or `groupBy`, the template literal produces empty strings that create blank lines in the output:

```js
return `select
    ${columnsToSQL(step.columns)}
from
    ${step.from}
${leftJoinClauses.join('\n')}   // empty string → blank line
${whereClause}                  // empty string → blank line
${groupByClause}`;              // empty string → blank line
```

## Goals

- CTE bodies indented inside their parentheses
- Multi-line column values aligned to their insertion depth
- No spurious blank lines from absent clauses
- No new line breaks introduced — helpers retain control over their own line break structure
- Indent size: 2 spaces

## Non-Goals

- Reformatting helper output (line breaks, internal structure) — helpers decide their own line break placement
- Adding an external SQL formatter dependency
- Changing any helper function signatures or return values
- Handling long single-line expressions (e.g., the `_item_list_attribution_row_id` expression)

## Solution Design

### Approach: Re-indent at insertion points (approach D)

The query builder already knows the indentation context at each insertion point. Two utility functions handle the re-indentation:

#### `reindent(sql, targetIndent)`

For multi-line column values. Normalizes continuation lines (lines after the first) to a target base indentation while preserving relative indentation within the fragment.

Algorithm:
1. Find the minimum indentation of non-empty continuation lines
2. Strip that minimum from each continuation line
3. Prepend `targetIndent` spaces to each continuation line

```js
const reindent = (sql, targetIndent) => {
    if (!sql.includes('\n')) return sql;
    const lines = sql.split('\n');
    const continuationLines = lines.slice(1).filter(l => l.trim());
    if (continuationLines.length === 0) return sql;
    const minIndent = Math.min(
        ...continuationLines.map(l => l.match(/^ */)[0].length)
    );
    const p = ' '.repeat(targetIndent);
    return lines[0] + '\n' + lines.slice(1)
        .map(l => l.trim() ? p + l.slice(minIndent) : '')
        .join('\n');
};
```

Example — `fixEcommerceStruct()` returns SQL with internal base indent of 4. When inserted at column depth 2 (inside a CTE at depth 4 after `indentBlock`), `reindent` strips the 4-space base and applies 2, then `indentBlock` shifts everything by 2 more:

```sql
-- before (helper's raw output pasted at column position)
    (select as struct ecommerce.* replace(
    if(ecommerce.transaction_id <> '(not set)', ...) as transaction_id,

-- after reindent + indentBlock
    (select as struct ecommerce.* replace(
      if(ecommerce.transaction_id <> '(not set)', ...) as transaction_id,
```

#### `indentBlock(sql, spaces)`

For CTE bodies. Shifts an entire SQL block right by prepending spaces to every non-empty line. Preserves all relative indentation.

```js
const indentBlock = (sql, spaces) => {
    const p = ' '.repeat(spaces);
    return sql.split('\n')
        .map(l => l.trim() ? p + l : '')
        .join('\n');
};
```

### Changes to `queryBuilder`

#### `columnsToSQL`

Apply `reindent` to each column entry before joining:

```js
// current: .join(',\n    ')
// new:     .join(',\n' + pad) where pad = ' '.repeat(INDENT)

.map(([key, value]) => {
    let entry;
    if (key === value) entry = value;
    else if (key.startsWith('[sql]')) entry = value;
    else entry = `${value} as ${key}`;
    return reindent(entry, INDENT);
})
.join(',\n' + pad);
```

#### `selectSQL`

Build parts array instead of a template literal with potentially empty interpolations:

```js
const selectSQL = (step) => {
    const parts = [`select\n${pad}${columnsToSQL(step.columns)}`];
    parts.push(`from\n${pad}${step.from}`);
    if (step.leftJoin) {
        step.leftJoin.forEach(join => {
            parts.push(`left join\n${pad}${join.table} ${join.condition}`);
        });
    }
    if (step.where) {
        parts.push(`where\n${pad}${reindent(step.where, INDENT)}`);
    }
    if (step.groupBy) {
        parts.push(`group by\n${pad}${step.groupBy.join(', ')}`);
    }
    return parts.join('\n');
};
```

#### CTE wrapping

Indent the `selectSQL` body inside the CTE parentheses:

```js
// current:
`${step.name} as (${selectSQL(step)})`
// joined with: ',\n    '

// new:
const body = indentBlock(selectSQL(step), INDENT);
`${step.name} as (\n${body}\n)`
// joined with: ',\n'
```

### How indentation flows

With `INDENT = 2`:

| Context | Keywords | Column values | Multi-line value continuation |
|---------|----------|---------------|-------------------------------|
| Top-level SELECT | indent 0 | indent 2 | indent 2 + relative |
| CTE body | indent 2 | indent 4 | indent 4 + relative |

The `reindent` step normalizes helper output to base indent 2. Then `indentBlock` shifts CTE bodies by an additional 2. The helpers' relative indentation (their internal line break structure) is preserved throughout.

```sql
    condition

    ...
```

**After:**
```sql
with event_data as (
  select
    col1,
    col2
  from

  where

session_data as (
  select

    event_data

  group by
    session_id

select
  ...
```

### Multi-line column value (`fixEcommerceStruct`)

**Before:**
```sql
    (select as struct ecommerce.* replace(
    if(ecommerce.transaction_id <> '(not set)', ecommerce.transaction_id, null) as transaction_id,
    if(
      event_name = 'purchase',
      coalesce(
        if(is_nan(ecommerce.purchase_revenue), null, ecommerce.purchase_revenue),

  )) as ecommerce,
```

**After:**
```sql
    (select as struct ecommerce.* replace(
      if(ecommerce.transaction_id <> '(not set)', ecommerce.transaction_id, null) as transaction_id,
      if(
        event_name = 'purchase',
        coalesce(
          if(is_nan(ecommerce.purchase_revenue), null, ecommerce.purchase_revenue),
          safe_cast(... as float64)
        ),
        null
      ) as purchase_revenue
    )) as ecommerce,
```

### Absent clauses (no blank lines)

**Before:**
```sql

    item_list_attribution

    _item_list_attribution_row_id
```

**After:**
```sql
from
  item_list_attribution

  _item_list_attribution_row_id
```

## Success Criteria

- [ ] CTE bodies are indented inside their parentheses
- [ ] Multi-line column values have continuation lines aligned to insertion depth
- [ ] No spurious blank lines from absent `where`/`groupBy`/`leftJoin` clauses
- [ ] Indent size is 2 spaces throughout
- [ ] No line breaks are added or removed — only indentation changes
- [ ] All existing tests pass (BigQuery dry-run validation)
- [ ] Assertion SQL generation is also correctly formatted (uses the same `queryBuilder` indirectly or follows the same patterns)

## Testing Strategy

### Existing tests

The test suite validates SQL via BigQuery dry-run. Since the change is whitespace-only (no semantic SQL changes), all existing tests should pass without modification. This is the primary validation.

### Visual inspection

Generate the full enhanced events SQL with a representative config (item list attribution enabled, session params, event params to columns) and visually verify the formatting. This was already done during prototyping — all tests passed.

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| `reindent` miscalculates `minIndent` for edge-case helper output | Low | Helpers mostly use consistent indentation; single-line values are no-ops. Dry-run tests catch syntax errors. |
| Assertions or other SQL generators outside `queryBuilder` have inconsistent formatting | Low | Assertion queries use string templates, not `queryBuilder`. They can adopt `reindent`/`indentBlock` independently if needed. |
